# Model Context Protocol — 2025-11-25

## 0. 본 MCP의 사용 범위

본 프로젝트는 **stdio JSON-RPC 서버**만 구현합니다. 클라이언트, HTTP/SSE transport, sampling, elicitation, resources subscription, prompts는 범위 밖. 필요한 서브셋:

- 라이프사이클: `initialize` → `notifications/initialized` → 작업 → 종료(transport close)
- 헬스체크: `ping`
- 도구: `tools/list`, `tools/call`, `notifications/tools/list_changed`(선택)
- 유틸리티 노티피케이션: `notifications/cancelled`, `notifications/progress`
- 표준 JSON-RPC 에러

## 1. 메시지 모델 (JSON-RPC 2.0 위)

모든 메시지는 JSON-RPC 2.0을 따릅니다. 세 종류:

### Request
```json
{
  "jsonrpc": "2.0",
  "id": "<string|integer, unique>",
  "method": "<string>",
  "params": { /* method-specific */ }
}
```

### Response (success)
```json
{
  "jsonrpc": "2.0",
  "id": "<matches request id>",
  "result": { /* method-specific */ }
}
```

### Response (error)
```json
{
  "jsonrpc": "2.0",
  "id": "<matches request id, or null if id couldn't be parsed>",
  "error": {
    "code": <integer>,
    "message": "<string>",
    "data": <any, optional>
  }
}
```

### Notification (one-way, no id, no response)
```json
{
  "jsonrpc": "2.0",
  "method": "<string>",
  "params": { /* method-specific, optional */ }
}
```

규칙:
- `id`는 request에 필수, notification에 금지.
- `id`는 string 또는 integer. 한 세션 내 unique.
- 알 수 없는 notification 수신 시 응답하지 말 것 (notification은 응답이 없는 것이 정의).

## 2. Stdio Transport — 프레이밍

MCP 사양상 stdio transport에서는 **줄바꿈으로 구분된 JSON**입니다 (LSP의 `Content-Length` 헤더 없음). 한 메시지 = 한 줄.

- 한 줄에 하나의 JSON object.
- UTF-8 인코딩.
- 메시지 자체에 raw newline 포함 금지(JSON 인코딩 시 `\n`으로 escape).
- stdout은 protocol 채널. stderr는 자유롭게 사용 가능 (로그·디버그).

## 3. 라이프사이클

### 3.1 Initialize 요청 (client → server)

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": { /* client capabilities */ },
    "clientInfo": {
      "name": "ExampleClient",
      "title": "Example Client",
      "version": "1.0.0"
    }
  }
}
```

### 3.2 Initialize 응답 (server → client)

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "tools": { "listChanged": true }
    },
    "serverInfo": {
      "name": "mcpswx",
      "title": "swiftmcp",
      "version": "0.0.1"
    },
    "instructions": "Optional usage hint shown to LLM"
  }
}
```

본 MCP의 응답 capabilities는 **`tools`만** 켭니다. 다른 카테고리(`resources`, `prompts`, `logging`, `tasks`)는 켜지 않음.

### 3.3 Initialized 노티피케이션 (client → server)

initialize 응답 처리 후 클라이언트가 발송. 이 시점부터 일반 요청을 받기 시작.

```json
{ "jsonrpc": "2.0", "method": "notifications/initialized", "params": {} }
```

### 3.4 ProtocolVersion 협상

- 서버는 클라이언트가 보낸 버전을 **그대로 받거나**, 자신이 지원하는 가장 가까운 버전을 응답에 적음.
- 양측이 합의 못 하면 양측 중 하나가 transport를 닫는다.
- 본 MCP는 `2025-11-25` 단일 버전만 지원. 다른 버전이 들어오면 그대로 `2025-11-25`로 응답하고, 클라이언트가 거부하면 자연 종료.

### 3.5 종료

stdio transport에서는 stdin EOF 또는 transport close가 종료 신호. 별도 shutdown 메시지 없음.

## 4. Ping

양측 누구나 보낼 수 있는 헬스체크.

요청:
```json
{ "jsonrpc": "2.0", "id": "abc", "method": "ping" }
```

응답:
```json
{ "jsonrpc": "2.0", "id": "abc", "result": {} }
```

받은 쪽은 **promptly** 빈 result로 응답. 본 MCP는 클라이언트의 ping에 즉시 응답만 하면 됨.

## 5. Tools

### 5.1 tools/list

요청:
```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list",
  "params": { "cursor": "<optional pagination cursor>" }
}
```

응답:
```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "get_weather",
        "title": "Weather Information Provider",
        "description": "Get current weather for a location",
        "inputSchema": {
          "type": "object",
          "properties": { "location": { "type": "string" } },
          "required": ["location"]
        },
        "icons": [
          { "src": "https://...", "mimeType": "image/png", "sizes": ["48x48"] }
        ],
        "execution": { "taskSupport": "optional" }
      }
    ],
    "nextCursor": "<optional, present if more pages>"
  }
}
```

- `name`: 호출 시 키.
- `title`: 사람이 보는 라벨 (선택).
- `description`: LLM이 도구 선택에 쓰는 텍스트 (필수에 가까움).
- `inputSchema`: JSON Schema. type=object 권장.
- `icons`, `execution`: 선택.
- `nextCursor`: 응답 페이지가 더 있으면 포함. 본 MCP의 도구 수가 적은 동안 사용 안 함.

### 5.2 tools/call

요청:
```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "location": "New York" }
  }
}
```

응답 (success):
```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      { "type": "text", "text": "Current weather: 72°F, partly cloudy" }
    ],
    "isError": false
  }
}
```

응답 (도구 실행 실패):
```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      { "type": "text", "text": "Failed to fetch weather: connection timeout" }
    ],
    "isError": true
  }
}
```

`content` 항목 타입:
- `text` — 일반 텍스트.
- `image` — base64 + mimeType.
- `resource_link` — 리소스 URI 참조 (큰 산출물에 권장).
- `audio`, `embedded_resource` — 사양 정의됨, 본 MCP는 사용 안 함.

본 MCP의 큰 산출물(SIL/AST/IR)은 `text` 안에 임시 파일 경로 + 요약을 담거나, `resource_link`로 노출. (현 단계에서는 단순 `text`로 시작.)

### 5.3 에러 채널 분리

| 케이스 | 채널 |
|---|---|
| 알 수 없는 도구 이름, 잘못된 인자 스키마 | **JSON-RPC error** (`code: -32602` 등) |
| 외부 프로세스 실행 실패, sandbox 거부, timeout | **Tool Result + `isError: true`** |
| 사용자 Swift 코드의 컴파일 진단(에러·워닝) | **Tool Result + `isError: false`**, content에 진단 (분석 도구의 결과는 진단 자체가 산출물) |

### 5.4 notifications/tools/list_changed (선택)

서버가 도구 목록을 변경했을 때 클라이언트에게 알림. 본 MCP는 도구 목록이 정적이므로 발송 안 함. capabilities에서 `listChanged: false`로 노출하면 더 정확하지만, `true`로 두고 발송 안 해도 클라이언트는 항상 polling 가능.

## 6. 유틸리티 노티피케이션

### 6.1 Cancellation

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": {
    "requestId": "<id of in-flight request to cancel>",
    "reason": "User requested cancellation"
  }
}
```

- 클라이언트가 진행 중인 요청을 취소하고 싶을 때 발송.
- 서버는 best-effort로 해당 요청 처리를 중단. 이미 응답을 보냈다면 무시.
- 본 MCP는 long-running 도구(격리 빌드/실행)에서 자식 프로세스 종료 신호로 매핑.

### 6.2 Progress

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progressToken": "<provided in original request meta>",
    "progress": 42,
    "total": 100,
    "message": "Compiling module 3 of 5"
  }
}
```

- 요청 시 `params._meta.progressToken`에 토큰을 넣으면, 서버가 해당 토큰으로 진행 상황을 노티피케이션으로 보낼 수 있음.
- 본 MCP는 Stage 1에서는 progress 미사용. Stage 2 이후 long-running에서 도입 검토.

## 7. JSON-RPC 표준 에러 코드

| code | name | 사용 |
|---|---|---|
| -32700 | Parse error | 받은 데이터가 유효한 JSON이 아님 |
| -32600 | Invalid Request | 메시지 envelope 형태 위반 |
| -32601 | Method not found | 우리가 모르는 method |
| -32602 | Invalid params | 인자 스키마 위반, 누락 필드 |
| -32603 | Internal error | 서버 내부 예외 (catch-all) |

서버 정의 가능한 추가 코드는 `-32000`부터 `-32099` 범위(Server error 예약). MCP 사양에 추가 표준 코드가 정의될 수 있으니 새 메서드 도입 시 사양 확인.

## 8. 본 MCP 매핑 (구현 가이드)

### 8.1 Codable Swift 타입 (본 MCP가 노출할 와이어 포맷)

```swift
public struct Request: Codable, Sendable {
    public let jsonrpc: String  // "2.0"
    public let id: ID
    public let method: String
    public let params: AnyCodable?
}

public struct Response: Codable, Sendable {
    public let jsonrpc: String  // "2.0"
    public let id: ID?          // null when parse error before id
    public let result: AnyCodable?
    public let error: ErrorObject?
}

public struct Notification: Codable, Sendable {
    public let jsonrpc: String  // "2.0"
    public let method: String
    public let params: AnyCodable?
}

public enum ID: Codable, Sendable, Hashable {
    case string(String)
    case integer(Int64)
}

public struct ErrorObject: Codable, Sendable {
    public let code: Int
    public let message: String
    public let data: AnyCodable?
}
```

`AnyCodable`은 직접 정의하거나, 메서드별로 typed `params`/`result`를 두는 두 갈래 중 선택. **typed가 안전** — 우리가 다루는 메서드 수가 적음.

### 8.2 Stdio loop 골격

```swift
// stderr 로그용 핸들 분리 (stdout은 protocol)
let stdin = FileHandle.standardInput
let stdout = FileHandle.standardOutput
let stderr = FileHandle.standardError

for try await line in stdin.bytes.lines {
    guard let data = line.data(using: .utf8) else { continue }
    try await handleIncoming(data)
}
```

각 라인을 디코드 → 메서드 라우팅 → 응답 직렬화 → stdout 한 줄 쓰기.

### 8.3 응답 라이트 단위

```swift
let encoded = try encoder.encode(response)
try stdout.write(contentsOf: encoded)
try stdout.write(contentsOf: Data([0x0a]))  // newline
```

`JSONEncoder`는 default로 raw newline을 포함하지 않음 (escaped로 출력). 줄바꿈은 우리가 한 번 더 붙임.

### 8.4 동시성 정책

- stdin 읽기는 단일 Task.
- 메서드 핸들러는 dispatch 후 각자 Task에서 실행.
- 응답 쓰기는 `actor`로 직렬화 (stdout은 한 번에 한 줄 단위로만 atomic).

## 9. Pitfalls

- **stdout 오염**: `print(...)`는 stdout으로 가서 protocol 메시지 사이에 plaintext가 끼어 클라이언트 파싱 실패. 절대 사용 금지. 디버그는 stderr.
- **메시지 단위 vs JSON 단위**: 한 줄 = 한 메시지. JSON 안에 raw newline 들어가면 깨짐 — encoder가 `\n`로 escape하는지 확인.
- **id 매칭**: 응답의 `id`는 request의 `id`와 정확히 같은 타입(string vs integer). 변환 금지.
- **알 수 없는 method**: notification이면 무시, request이면 `-32601`.
- **알 수 없는 notification**: 무시. 응답 보내지 말 것.
- **id 없는 request**: `-32600` Invalid Request로 응답.
- **parse error 시 id**: id 추출이 안 되면 `id: null`로 에러 응답.
- **batch**: JSON-RPC 2.0의 batch(배열 형태)는 MCP 사양상 사용하지 않음. 본 MCP는 단일 메시지만 처리.
- **서버 발송 ping**: 본 MCP는 서버측 ping을 발송하지 않음 — 클라이언트 ping에 응답만.
